### 4.1 Function Rules
- **50-line hard limit per function** ??if a function exceeds 50 lines, refactor immediately
- One function = one responsibility. If "and" appears in its description, split it.
- Extract validation logic into `validate_<n>()` functions
- Extract transformation logic into `transform_<n>()` functions
- Extract all I/O (file, network, DB) into `io_<n>()` functions
- No business logic mixed with I/O logic
- **Every function must be declared in Blueprint Section 4 before it is implemented**

### 4.2 Naming Conventions
- Functions: `snake_case`
- Classes: `PascalCase`
- Constants: `UPPER_SNAKE_CASE`
- Files: `snake_case.ts`
- Test functions: `test_<function_name>_<scenario>()`

### 4.3 Docstrings (required on every function)
```typescript
/**
 * Single-line purpose.
 *
 * @param param - description
 * @returns description
 * @throws ErrorType - when condition
 * @traces FR-NNN, AC-NNN
 */
function function_name(param: type): ReturnType { ... }
```

### 4.4 Error Handling
- Never use bare `catch(e)` without specific handling
- Always catch specific error types where possible
- Every function that can fail must return a typed error or throw a documented exception
- No silent failures ??log or throw, never swallow

### 4.5 Code Quality
- No hardcoded values ??use constants or environment variables
- No unused imports
- No commented-out code in final output
- Type hints required on all function signatures
- **No import may be added that is not on the Blueprint import allowlist**
